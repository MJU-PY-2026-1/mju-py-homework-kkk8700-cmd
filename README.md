최종적으로 이 프로그램은 대학생의 학업 관리를 돕는 실용적인 일정 관리 소프트웨어로 완성될 계획입니다.

## 🟩 [2차 과제: V1.0] 입출력 + 리스트 + 조건문 - 향후 작성 예정
### **✨2차 과제 내용:**
✨ 2차 과제 내용:

이번 과제는 대학생 과제·시험 일정 관리 프로그램 구현으로, 사용자로부터 여러 일정 정보를 입력받아 리스트에 저장하고, 남은 일수와 중요도, 예상 소요 시간을 바탕으로 일정의 우선순위를 판정하는 시스템을 구축하는 것입니다. 특히 if-elif-else 조건문과 and/or 논리 연산자를 활용하여 일정의 긴급도와 특별 관리 대상 여부를 구분하는 기능을 포함하도록 설계했습니다. 또한 리스트 메소드와 내장함수를 활용하여 여러 일정 데이터를 저장·출력·삭제·분석할 수 있도록 구성했습니다.

input()으로 과목명, 일정 종류(과제/시험), 일정 이름, 남은 일수, 중요도, 예상 소요 시간을 입력받아 하나의 일정 정보를 생성.
입력받은 값 중 남은 일수와 중요도는 int(), 예상 소요 시간은 float()로 형변환하여 계산에 활용.
하나의 일정 정보를 리스트로 만들고, 이를 전체 일정 리스트에 append()하여 최소 3개 이상의 데이터를 저장.
if-elif-else를 사용하여 일정 우선순위를 긴급 / 보통 / 여유 단계로 판정하는 로직 구현.
and 연산자를 활용하여 남은 일수가 매우 적고 중요도도 높은 일정은 ‘최우선 관리 일정’으로 추가 분류하는 기능 구현.
len(), sum(), max(), remove() 등을 활용하여 일정 개수 확인, 중요도 합계 계산, 가장 중요한 일정 확인, 일정 삭제 기능을 구현.
+= 복합 대입 연산자를 사용하여 전체 예상 공부 시간을 누적 계산하는 기능 추가.

적용 내용: AI와의 협업을 통해 프로그램을 기초 입력 및 저장 단계 → 우선순위 판정 단계 → 반복문 기반 메뉴 시스템 단계 → 과목별 세부 조회 단계로 확장할 수 있는 로드맵을 정리함. 이를 통해 이번 2차 과제에서는 핵심 로직 구현에 집중하고, 이후 과제에서는 반복문을 이용한 메뉴 반복 실행과 탐색 기능까지 발전시킬 수 있도록 방향을 설정함.
🛠️ Troubleshooting & 기술 회고:

1. 문제 1: 숫자를 입력했는데 계산이 되지 않는 현상 (TypeError)

원인: input()으로 입력받은 값은 기본적으로 문자열이기 때문에, 남은 일수나 중요도, 예상 소요 시간을 바로 계산에 사용하면 오류가 발생했음.
해결: 남은 일수와 중요도는 int(), 예상 소요 시간은 float()를 사용해 형변환한 뒤 계산하도록 수정하여 해결함.

2. 문제 2: 조건문은 맞는 것 같은데 일정이 이상하게 분류되는 문제

원인: if-elif-else 조건의 순서가 적절하지 않아 긴급한 일정이 보통 일정으로 처리되는 경우가 있었음.
해결: 가장 우선순위가 높은 조건을 먼저 검사하도록 순서를 재배치하여, 긴급한 일정부터 정확히 판정되도록 수정함.

3. 문제 3: 여러 일정 데이터를 저장할 때 어떤 형태로 묶어야 할지 헷갈리는 문제

원인: 과목명, 일정 종류, 일정 이름, 남은 일수 등을 각각 따로 저장하면 서로 어떤 데이터가 한 묶음인지 구분하기 어려웠음.
해결: 하나의 일정을 [과목명, 일정종류, 일정이름, 남은일수, 중요도, 예상소요시간] 형태의 리스트로 만들고, 이 일정들을 다시 큰 리스트에 저장하는 방식으로 구조를 정리하여 해결함.

4. 문제 4: 리스트 메소드와 내장함수를 과제 조건에 맞게 자연스럽게 넣기 어려운 문제

원인: 과제 조건상 append() 외에도 여러 메소드와 내장함수를 사용해야 했기 때문에, 일정관리 프로그램 안에서 이를 억지스럽지 않게 적용할 필요가 있었음.
해결: 일정 추가에는 append(), 일정 삭제에는 remove(), 일정 개수 확인에는 len(), 중요도 합계 계산에는 sum(), 가장 중요한 일정 확인에는 max()를 연결하여 프로그램 기능 안에 자연스럽게 반영함.
    
### **🛠️ Troubleshooting & 기술 회고:**
(예시) 
1. **문제 1: 숫자를 입력했는데 계산이 안 되는 현상 (TypeError)**
    * **원인:** `input()`으로 받은 데이터는 기본적으로 '글자'라서 숫자와 곱셈 연산이 불가능했음.
    * **해결:** `int()`와 `float()` 함수를 사용하여 입력값을 숫자로 바꾸는 '형변환' 마법을 부려 해결함.
2. **문제 2: 조건문 뒤에 자꾸 생기는 빨간 줄 (SyntaxError)**
    * **원인:** `if`와 `elif` 조건식 끝에 마법의 기호인 `:`(콜론)을 자꾸 빠뜨려서 발생함.
    * **해결:** 파이썬의 모든 조건문과 반복문 끝에는 항상 땡땡(`:`)이 들어가야 한다는 규칙을 학습하고 수정함.

이번 3차 과제는 기존 2차 과제에서 구현한 대학생 과제·시험 일정 관리 프로그램을 발전시켜, 사용자가 원하는 기능을 반복적으로 선택할 수 있는 메뉴형 프로그램으로 확장하는 것입니다.
while True 무한 루프를 사용하여 프로그램 실행 시 메인 메뉴가 계속 출력되도록 구성하고, 사용자가 종료 메뉴를 선택했을 때만 break로 프로그램이 끝나도록 구현할 예정입니다.
또한 2차 과제에서 작성했던 일정 입력, 전체 일정 조회, 우선순위 분석 기능을 각각 개별 함수(def) 로 분리하여 코드 구조를 더 보기 좋고 체계적으로 만들 것입니다.
최소 1개 이상의 함수는 매개변수를 받아 처리하도록 설계하고, 최소 1개 이상의 함수는 계산 결과를 return 하도록 구성하여 함수 활용 능력을 보여줄 계획입니다.
전역 리스트에는 전체 일정 정보를 저장하고, 함수 내부에서는 지역 변수를 사용하여 역할을 구분할 예정이며, 일정 추가 시 전체 공부 시간 갱신과 같은 부분에서 global 키워드도 1회 이상 활용할 계획입니다.
최종적으로 이 프로그램은 1. 일정 입력 / 2. 전체 조회 / 3. 일정 분석 / 4. 일정 삭제 / 5. 종료 형태의 메뉴를 가진 실용적인 학업 일정 관리 프로그램으로 완성할 예정입니다.
🤖 AI 파트너십 과정
1. 내용 1
프롬프트 요약:
“2차에서 만든 대학생 일정관리 프로그램을 3차 과제 조건에 맞게 while True 메뉴형 프로그램으로 확장하려면 어떤 구조로 바꾸는 게 좋은지 정리해줘.”
적용 내용:
AI와의 대화를 통해 기존 2차 과제 코드를 단순 실행형이 아니라 메뉴 선택형 구조로 바꾸는 방향을 정리하였다. 이에 따라 프로그램 시작 시 메인 메뉴를 출력하고, 사용자가 번호를 입력하면 일정 입력, 조회, 분석, 삭제 기능을 선택적으로 실행하도록 설계하였다. 특히 while True를 사용하고 종료 메뉴에서만 break가 실행되도록 하여 3차 과제의 핵심 요구사항을 만족하도록 방향을 잡았다.
2. 내용 2
프롬프트 요약:
“입력, 조회, 분석 기능을 함수로 나누고 싶은데 3차 과제 기준에 맞게 최소 3개 함수로 자연스럽게 분리하는 방법을 알려줘.”
적용 내용:
AI의 도움을 받아 프로그램 기능을 일정 입력 함수, 전체 일정 출력 함수, 우선순위 분석 함수, 일정 삭제 함수처럼 역할별로 분리하는 구조를 구상하였다. 이를 통해 메인 로직 안에서는 함수 호출 중심으로 코드가 간결하게 보이도록 정리할 수 있었고, 3차 과제 평가 기준인 “메인 로직은 목차처럼 구성”이라는 방향에도 맞출 수 있었다.
3. 내용 3
프롬프트 요약:
“매개변수와 return, global을 일정관리 프로그램 안에서 억지스럽지 않게 넣으려면 어떤 기능에 적용하면 좋을까?”
적용 내용:
AI와 협업하여 예를 들어 상태 판정 함수는 남은 일수와 중요도를 매개변수로 받아 긴급/보통/여유 값을 return 하도록 설계하고, 전체 공부 시간이나 전체 일정 목록처럼 프로그램 전체에서 공통으로 사용하는 데이터는 전역변수로 두되, 일정 추가 시 갱신하는 부분에서 global을 한 번 사용하는 방식으로 구조를 정리하였다. 이 과정에서 전역변수와 지역변수의 차이도 함께 이해할 수 있었다.
🛠️ Troubleshooting & 기술 회고:
1. 문제 1: while 메뉴를 잘못 만들면 프로그램이 끝나지 않거나 의도치 않게 종료되는 문제
원인:
while True를 사용하더라도 종료 조건과 break 위치를 잘못 두면 메뉴가 계속 반복되거나, 반대로 원하는 기능 실행 전에 프로그램이 종료될 수 있다.
해결:
메인 메뉴에서 사용자의 선택을 먼저 입력받고, if-elif-else로 기능을 나눈 뒤 종료 번호를 선택했을 때만 break 되도록 구조를 정리하였다. 이를 통해 프로그램 흐름이 더 명확해졌다.
2. 문제 2: 함수로 분리했을 때 변수들이 서로 연결되지 않는 문제
원인:
2차에서는 한 번에 작성하던 코드를 3차에서 함수로 나누면서, 어떤 변수는 함수 안에서만 쓰이고 어떤 변수는 전체 프로그램에서 공유되어야 하는지 구분이 헷갈렸다.
해결:
일정 리스트처럼 프로그램 전체에서 공통으로 사용하는 값은 전역변수로 두고, 함수 내부에서만 잠깐 쓰는 값은 지역변수로 구분하였다. 또한 꼭 필요한 경우에만 global을 사용하도록 정리하여 변수 관리 구조를 명확하게 만들었다.
3. 문제 3: 2차 기능을 3차 요구사항에 맞게 자연스럽게 확장하는 문제
원인:
2차에서는 일정 3개를 한 번 입력하고 끝나는 구조였기 때문에, 이를 메뉴형 프로그램으로 바꾸려면 입력·조회·분석 기능을 다시 묶어 재구성할 필요가 있었다.
해결:
입력 기능은 사용자가 메뉴에서 선택할 때마다 새 일정을 추가할 수 있게 바꾸고, 조회와 분석은 저장된 리스트를 바탕으로 언제든 다시 실행할 수 있게 함수화하였다. 이 과정에서 2차 프로그램이 단순 실습형에서 실제 관리형 프로그램으로 발전했다는 점을 확인할 수 있었다.
